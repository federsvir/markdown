##Спецификация требований
###Введение
*Представлены требования, типы пользователей и заинтересованные лица к системе “сайт fpizza”*
####Описание системы
#####Назначение
Система предназначена для удобного выбора пользователями необходимой для заказа пиццы, последующей оплаты заказа и отслеживания пользователем статуса заказа
######Границы системы и бизнес-требования
Система должна обеспечить:
+ Предоставление пользователю меню доступной для заказа пиццы с выбором доступных размера и топпинга для каждой
+ Предоставление пользователю способ оплаты заказа (банковской картой/онлайн кошельком/наличными, до/после доставки)
+ Получение статус заказа из логистического хаба
+ Возможность авторизации для последующего упрощения формирования заказа для пользователей
#######Краткое описание контекста использования
Система взаимодействует с внешними системами:
+ Получает данные пользователей из CRM-системы посредством её SOAP API
+ Получает данные заказов, товаров и остатков из ERP-системы посредством файлового обмена из общего FTP-сервера.
+ Предоставляет информацию о звонках пользователя CRM- и ERP-системами посредством REST API.
########Заинтересованные лица
+ *Внутренние ЗЛ:*
+ *Программисты сайта*
+ *Дизайнеры сайта*
+ *Системные аналитики*
+ *Внешние ЗЛ:*
+ *Клиенты*
#########Типы пользователей
Регулярные пользователи - лица, которым за их активность предоставляются льготы
Нерегулярные пользователи - оставшийся круг лиц, чья активность является ниже пороговой
###########Основные функции
+ Я как пользователь хочу иметь возможность выбора пиццы из ассортимента с возможностью выбора размера и топпинга, чтобы выбрать именно то, что мне нужно
+ Я как пользователь, хочу отслеживать статус своего заказа, чтобы знать, что с ним происходит
+ Я как пользователь, хочу иметь возможность выбора способа оплаты заказа, чтобы оплатить его так, как мне удобней
###########Ограничения
+ Пользователи будут использовать систему, используя канал связи низкой пропускной способности.
+ При разработке системы запрещается использовать open-source-компоненты.
+ Система должна хранить данные пользовательских действий в течение 30 дней по требованиям федерального закона.
############Требования
+ Система должна предоставлять пользователю меню доступной для заказа пиццы с выбором доступных размера и топпинга для каждой
+ Система должна предоставлять пользователю способ оплаты заказа (банковской картой/онлайн кошельком/наличными, до/после доставки)
+ Система должна получать статус заказа из логистического хаба и предоставлять его пользователю
+ Система должна предоставлять возможность авторизации для последующего упрощения формирования заказа для пользователей
+ Система должна хранить данные о заказах
+ Система должна уметь отделять регулярных пользователей от нерегулярных
##############Сценарии использования и функциональные требования
Нерегулярные пользователи: Для нерегулярных пользователей система должна предоставлять стандартное меню выбора пиццы, с последующей возможностью оплаты заказа различными способами
Регулярные пользователи: Для регулярных пользователей система должна вдобавок к стандартному меню предоставлять меню предпочтений и рекомендаций. При оплате должны предоставляться льготы
###############Логическая модель данных
*Сущности:*
*Сайт (название, ссылка)*
*Пицца (название, цена, размер)*
*Корзина*
*Заказ (компоненты, стоимость, статус)*
*Пользователь (имя, номер телефона)*
*Курьер (имя, номер телефона)*
*Адрес доставки*
*Связи:*
*Пользователь добавляет пиццу с сайта в корзину*
*Адрес доставки связан с заказом*
*Курьер доставляет заказ по адресу доставки*
################Пользовательские интерфейсы
Две страницы: выбор и добавление нужных товаров в корзину, а также страница оплаты заказа. Также на первой странице можно авторизоваться и просмотреть статус оформленного заказа

#################Удобство использования
Для постоянных пользователей система должна предоставлять при входе набор список товаров из предыдущих заказов для упрощения укомплектации текущего заказа
Пользовательский интерфейс мобильного приложения должен соответствовать требованиям рекомендаций Apple к iOS-приложениям
Цвета и шрифты пользовательского интерфейса веб-сайта должны соответствовать требованиям стандарта по инклюзивному дизайну WCAG 2.0
##################Атрибуты качества
Система должна обрабатывать 95% запросов от пользователей не дольше 1 секунду.
Система должна обеспечивать доступность на уровне 99,9% времени работы.
